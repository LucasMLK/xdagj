问题描述
================================================================================

观察到的现象:
  ✅ 接受: 43 个区块
  ❌ 拒绝: 150 个区块
  ─────────────────
  拒绝率: 77.6%

用户质疑: 这个拒绝率是否过高？是否存在问题？

================================================================================
深入调查结果
================================================================================

1. Pool配置
   • initialDifficulty = 1000 ← 这是根本原因！
   • 区块实际难度: 109-342
   • Pool难度相对过低

2. Miner行为
   • 提交频率: 25-100 shares/秒
   • 64秒内提交: 约1600-6400个shares
   • 这是极高的频率！

3. Candidate Block机制
   • Pool每64秒请求1次新candidate
   • 一旦有区块被接受，candidate立即更新
   • 基于旧candidate的shares被拒绝

================================================================================
时间线示例 (64秒Epoch内)
================================================================================

T=0s     │ Pool获取Candidate A (epoch=100)
         │
T=0-64s  │ Miner疯狂挖矿
         │ ├─ share #1, #2, #3, ... #2000
         │ └─ 其中约10个符合区块难度
         │
T=10s    │ Share #1符合区块难度
         │ Pool提交给Node
         │ ✅ ACCEPTED → 成为区块
         │ ⚠️  Candidate更新为B
         │
T=12s    │ Share #2符合区块难度（基于Candidate A）
         │ Pool提交
         │ ❌ REJECTED - "unknown block (not based on our candidate)"
         │
T=15s    │ Suite2的区块通过P2P到达
         │ ✅ ACCEPTED
         │ ⚠️  Candidate更新为C
         │
T=18s    │ Share #3符合区块难度（基于Candidate A）
         │ ❌ REJECTED - 已经过时2代
         │
T=20s    │ Share #4
         │ ❌ REJECTED
         │
...      │ 持续到T=64s
         │ 剩余所有shares被拒绝

结果: 1个接受，9个拒绝 = 90%拒绝率

================================================================================
数学模型
================================================================================

设：
  N = 64秒内Miner提交的shares总数 ≈ 2000
  M = 其中符合区块难度的shares数 ≈ 10
  R = Suite2节点产生的区块数 ≈ 2-3

则：
  第1个share被接受 → 剩余 M-1 个基于旧candidate
  Suite2产生 R 个区块 → 再次更新candidate
  
  最终接受数 = 1 + 偶尔运气好的几个 ≈ 2-3
  最终拒绝数 = M - 接受数 ≈ 7-9
  
  拒绝率 = 7-9 / 10 ≈ 70-90%

实际测试: 77.6% ✓ 符合预期

================================================================================
问题性质判断
================================================================================

❌ 不是Bug
✅ 是配置问题

原因：
  1. Pool difficulty设置过低 (1000 vs 实际区块难度109-342)
  2. 导致Miner提交过多shares
  3. 大量shares在提交前就已过时
  4. Pool和Node之间产生大量无效通信

影响：
  • CPU占用高 (Pool需处理大量shares)
  • 网络带宽浪费
  • 但不影响功能正确性
  • 数据最终一致性完美
