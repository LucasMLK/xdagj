五、问题发现与解决
================================================================================

【问题1】初始测试遇到数据库一致性问题

  现象:
    • 节点重启后，新区块被拒绝
    • 错误信息: "Link target not found: 0xfe365c205563f6..."
    • 该link target是上次测试的最后一个区块

  原因:
    • 数据库保留了旧数据
    • 新区块尝试链接到旧的候选区块
    • 节点无法找到该候选区块

  解决方案:
    • 清空devnet数据库目录
    • 从创世块重新开始测试
    ✅ 问题解决

【问题2】创世Epoch与当前Epoch差距过大导致无法挖矿

  现象:
    • canMine = false
    • miningStatus = "blocked"
    • 错误信息: "Large epoch gap detected (lag: 5646 epochs)"

  原因:
    • 创世配置的epoch: 27555273
    • 当前真实epoch: 27560900+
    • Epoch差距: 5646 (远超过miningMaxReferenceDepth=16)

  解决方案:
    • 用户正确识别了问题根源
    • 节点启动后自动同步到最新epoch
    • 等待epochGap降低到16以内
    ✅ 节点自动解决（约15秒后canMine=true）

六、结论与建议
================================================================================

【结论】

1. nonceRange优化效果显著
   ✅ 将Miner nonceRange从10000提高到1000000是最有效的优化
   ✅ Share提交频率降低99.8%，从50-100次/秒降至0.5次/分钟
   ✅ 系统负载大幅降低，预计减少50%以上

2. Pool难度优化效果有限
   ⚠️ 仅提高Pool difficulty，效果不明显
   ⚠️ 必须同时优化Miner nonceRange才能达到预期效果

3. 区块拒绝率大幅改善
   ✅ 从77.6%降至0%（阶段3测试中）
   ⚠️ 但测试时间较短（8分钟），需要长期观察

4. 系统稳定性良好
   ✅ 双节点系统运行稳定
   ✅ P2P同步正常
   ✅ 区块生成和验证机制正确

【生产环境建议】

1. Miner配置（推荐）
   nonceRange = 1000000  # 或更高（根据硬件性能调整）
   threads = 2-4         # 根据CPU核心数调整
   algorithm = "auto"

2. Pool配置（推荐）
   initialDifficulty = 50000    # 或更高（根据网络算力调整）
   blockInterval = 64s
   dynamicDifficulty = true     # 如果支持动态难度调整

3. 性能监控指标
   • Share提交频率: 目标 0.1-1次/分钟
   • 区块拒绝率: 正常范围 10-30%
