## 根本原因分析

### 原因1: BUG-STORAGE-002期间的数据丢失

**时间范围**: 节点首次启动 ~ 10:44（修复部署）

**机制**:
1. Blocks写入MemTable（内存）
2. 只有WAL被sync，MemTable未flush
3. 节点重启时：
   - MemTable内容丢失
   - WAL replay但blocks无height（因为height=0）
   - 只有已compaction到SST的blocks（如Genesis）保留

**结果**: Heights 1-93期间的大量blocks在节点重启时丢失

### 原因2: 两个节点的不同重启历史

**Node1和Node2的重启时间不同**，导致：
- Node1在某次重启后丢失了部分blocks（例如heights 91）
- Node2在另一次重启后丢失了另一部分blocks（例如heights 74, 75, 77等）
- 两个节点独立丢失了不同的历史数据

**证据**: 用户提供的对比表显示：
```
Node1有但Node2没有: 74, 75, 77, 79, 82, 84, 86, 87, 89, 90
Node2有但Node1没有: 91
两者都没有: 大量heights（3, 5-10, 13, 15, 18-20等）
```

### 原因3: 修复后的缓冲期和同步时间

**时间**: 10:44（修复）~ 11:45（height 94稳定）

**持续时间**: 约61分钟

**在此期间**:
1. **10:44-11:24 (40分钟)**:
   - 修复刚部署，节点需要时间同步
   - 旧的数据丢失问题仍可能影响某些epoch
   - Height 75 (epoch 27567191)仍有一次"Link target not found"错误

2. **11:24-11:45 (21分钟)**:
   - 数据逐渐稳定

3. **11:45之后**:

   - 不再有数据丢失
