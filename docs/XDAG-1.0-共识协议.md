# XDAG 1.0 共识协议

**版本**: 1.0
**状态**: 草稿
**最后更新**: 2025-11-21

## 摘要

XDAG 1.0 是一个基于时间段（epoch）的有向无环图（DAG）区块链，结合了工作量证明（PoW）共识与独特的区块选择机制。与传统区块链中区块形成线性链条不同，XDAG 允许多个区块在同一时间窗口（epoch）内竞争，胜者由哈希值决定。这种设计在保持去中心化和安全性的同时，实现了高交易吞吐量。

---

## 1. 核心概念

### 1.1 时间系统与时间段（Epoch）

XDAG 使用独特的时间表示系统：

#### 时间单位

**基本时间单位**: XDAG 使用 **1/1024 秒**作为基本时间精度（而非标准的 1/1000 秒）

**XDAG Epoch（完整时间戳）**:
- 使用 1/1024 秒精度的时间值
- 例如：1552800481279
- 转换公式：`xdag_epoch = (milliseconds * 1024) / 1000`
- 这是XDAG内部使用的完整时间戳

**Epoch Number（时间段编号）**:
- 代表一个64秒周期的编号
- 例如：23693854
- 计算公式：`epoch_number = xdag_epoch >> 16`（右移16位）
- 这是区块存储和竞争使用的时间段编号

#### 时间段特性

**一个 Epoch Number 的时长**:
```
时长 = 2^16 * (1/1024 秒) = 65536 / 1024 = 64 秒
```

**关键计算**:
- 从毫秒获取当前epoch number：`(ms * 1024 / 1000) >> 16`
- 从epoch number获取起始时间：`epoch_number << 16`
- 从epoch number获取结束时间：`(epoch_number << 16) | 0xffff`

**用途**:
- 将区块分组到64秒的竞争轮次中
- 每个epoch number内的所有候选区块进行竞争
- 创世epoch number由网络配置定义（主网/测试网/开发网）

**示例**:
```
毫秒时间戳:     1552800481279
↓ (ms * 1024 / 1000)
XDAG Epoch:     1590068716382
↓ (>> 16)
Epoch Number:   24261
```

### 1.2 区块生产与类型

#### 区块生产时机

**关键规则**: XDAG 的区块生产发生在**每个 epoch 结束的时间点**

- **生产时刻**: 当 epoch 时间片到达结束时刻（`epoch | 0xffff`），所有接入网络的节点都可以开始出块
- **Epoch 统一性**: 同一个 epoch 周期内，所有节点出块的 Block 的 epoch 编号都是**完全相同的**
- **并发竞争**: 多个节点可以在同一 epoch 结束时刻同时产生区块，这些区块将进入 epoch 竞争

**示例**:
```
Epoch 23693854 结束时刻: XdagTime.epochNumberToMainTime(23693854)
- 节点 A 出块: epoch = 23693854, hash = 0x0000012345...
- 节点 B 出块: epoch = 23693854, hash = 0x0000056789...
- 节点 C 出块: epoch = 23693854, hash = 0x0000098765...

这三个区块的 epoch 编号完全相同，将进入竞争
```

#### 区块类型

XDAG 1.0 中的所有区块都是参与 epoch 竞争的**候选区块**：

- **候选区块（Candidate Block）**: 在一个 epoch 结束时刻挖出的任何区块
- **主块（Main Block，胜者）**: epoch 中哈希值最小（算力最大）的候选区块
- **孤块（Orphan Block，败者）**: epoch 中的其他所有候选区块
- **创世区块（Genesis Block）**: 第一个区块（epoch 0, difficulty = 1）

**重要说明**:
1. 区块类型之间没有结构上的区别，类型由共识规则决定
2. 为了防止 1000 个节点保存 1000 个候选块，每个 epoch 周期内，**只保留算力最大的 16 个候选区块**
3. 算力第一（哈希值最小）的候选区块成为主块

### 1.3 区块结构

#### 区块头（Header，固定大小）
- **Epoch 编号**: 8 字节（属于哪个 64 秒窗口）
- **难度目标**: 32 字节（PoW 验证阈值）
- **随机数（Nonce）**: 32 字节（PoW 解）
- **Coinbase 地址**: 20 字节（挖矿奖励接收者）
- **哈希值**: 32 字节（缓存的区块哈希）

#### 链接（Links，可变大小）
一个区块包含对其他区块和交易的**引用**：

- **区块链接（Block Links）**: 对上一个 epoch 候选区块的引用
  - **最大数量**: 16 个区块引用（`MAX_BLOCK_LINKS`）
  - **用途**: 连接到上一个 epoch 的顶级候选区块

- **交易链接（Transaction Links）**: 对待处理交易的引用
  - **最大数量**: 1,484,984 个交易引用
  - **当前限制**: 1,024 个交易（保守的初始部署）
  - **用途**: 将用户交易纳入区块链

**总链接容量**: 每个区块最多 1,485,000 个链接（`MAX_LINKS_PER_BLOCK`）

#### 区块元数据（不序列化）
- **高度（Height）**: 主链中的顺序编号
- **累积难度**: 从创世区块开始的所有难度总和
- **状态**: MAIN（主块）、ORPHAN（孤块）或 PENDING（待处理）

## 2. 工作量证明（PoW）

### 2.1 哈希计算

区块哈希计算方式：
```
hash = Keccak256(epoch || difficulty || nonce || coinbase || links_count || link[0] || link[1] || ...)
```

其中：
- `epoch`: 8 字节
- `difficulty`: 32 字节
- `nonce`: 32 字节
- `coinbase`: 20 字节
- `links_count`: 4 字节（链接数量）
- 每个 `link[i]`: 33 字节（类型 + 哈希）

### 2.2 PoW 验证

如果满足以下条件，则区块有效：
```
hash ≤ baseDifficultyTarget
```

其中：
- `hash`: 区块的 Keccak256 哈希（解释为 UInt256）
- `baseDifficultyTarget`: 全网难度阈值
- 哈希值越小 = 计算工作量越大

### 2.3 初始难度目标

**主网/测试网**:
```
baseDifficultyTarget = 2^192
```
- 要求哈希值中约有 8 个零字节
- 平均挖矿时间：1 GH/s 约 5 小时/块
- 网络效应（1000 矿工 @ 10 GH/s）：约 346 块/epoch

**开发网**:
```
baseDifficultyTarget = 2^256 - 1（最大值）
```
- 接受任何区块（无需 PoW）
- 用于开发和测试

## 3. 基于 Epoch 的共识

### 3.1 Epoch 内竞争

#### 区块生产规则

**生产时机**: 每个 epoch 结束的时间点
- 所有接入网络的矿工节点都可以在 epoch 结束时刻出块
- 同一 epoch 内产生的所有区块，其 epoch 编号**完全相同**
- 多个节点并发产生的区块进入同一 epoch 的竞争池

**有效性验证**:
- 所有满足 `hash ≤ baseDifficultyTarget` 的候选区块都是有效的
- PoW 验证确保每个区块都有足够的计算工作量

#### 候选区块存储策略

**存储限制**: 为了防止存储无限增长（如 1000 个节点产生 1000 个候选块）
- 每个 epoch 周期内，**只保留算力最大的 16 个候选区块**
- 算力计算：`work = MAX_UINT256 / hash`（哈希值越小，算力越大）
- 超过 16 个的候选区块将被丢弃

**保留策略**:
1. 收集该 epoch 内所有有效候选区块
2. 按算力（work）降序排序
3. 保留前 16 个算力最大的候选区块
4. 丢弃其余候选区块

#### 胜者选择

**主块确定规则**:
```
主块（胜者）= epoch 中算力最大的候选区块
           = epoch 中 hash 值最小的候选区块
```

**理由**: `work = MAX_UINT256 / hash`，因此哈希值越小，算力越大

**示例**:
```
Epoch 23693854 结束时刻，1000 个节点同时出块:
  - 区块 A: hash = 0x0000012345... work = 1.23e+73 (胜者 - 最小哈希，最大算力)
  - 区块 B: hash = 0x0000056789... work = 1.18e+73 (孤块，Top 16)
  - 区块 C: hash = 0x0000098765... work = 1.15e+73 (孤块，Top 16)
  ...
  - 区块 P: hash = 0x0000999999... work = 1.02e+73 (孤块，Top 16)
  - 区块 Q, R, ..., Z: (丢弃，未进入 Top 16)

保留: 区块 A-P (16 个算力最大的)
丢弃: 区块 Q-Z 及其他 (984 个算力较小的)
```

### 3.2 主链选择

**主链**由累积难度决定：

```
主链 = 累积难度最大的链
```

其中：
- `累积难度` = 从创世区块到当前的所有主块难度之和
- 每个主块将其 `difficulty` 加到累积总和中
- 遵循"最重链获胜"原则（类似比特币）

**难度计算**:
```
区块难度 = MAX_UINT256 / hash
```
- 哈希值越小 → 难度越大
- 激励矿工寻找更小的哈希值

### 3.3 高度分配

主块获得顺序高度：
- 创世区块：height = 1
- 下一个主块：height = 上一高度 + 1
- 孤块：height = 0（不属于主链）

**示例**:
```
Epoch    主块        高度
-----------------------------
0        创世块       1
1        区块 A1      2
2        区块 B1      3
3        (无胜者)     -
4        区块 D1      4
```

## 4. 挖矿策略

### 4.1 引用选择

创建新候选区块时（在 epoch 结束时刻），矿工必须引用：

**1. 上一个 Epoch 的 Top 16 候选区块**（`MAX_BLOCK_LINKS`）:
- 获取上一个主块所在的 epoch
- 收集该 epoch 中保留的所有候选区块（最多 16 个）
- 按算力（work = MAX_UINT256 / hash）降序排序
- 选择前 16 个候选区块（如果不足 16 个，则全部选择）
- 向新候选区块添加这些区块链接

**理由**:
- 确保新区块连接到上一个 epoch 的最强候选区块
- 由于每个 epoch 只保留 Top 16 候选区块，这些都是算力最大的区块
- 主块（算力第一）必然在这 16 个中
- 给算力靠前的孤块一个被引用的机会
- 维护 DAG 结构完整性

**2. 交易池中的待处理交易**（最多 1,024 个）:
- 从交易池查询待处理交易
- 按交易费用排序（最高优先）
- 选择最多 1,024 个交易
- 向候选区块添加交易链接

**总链接数**: 16 个区块链接 + 最多 1,024 个交易链接 = 最多 1,040 个链接/块

### 4.2 挖矿限制

**引用深度限制**:
```
当前_epoch - 上一主块_epoch ≤ MINING_MAX_REFERENCE_DEPTH (16 epochs)
```

**目的**: 防止过时节点用陈旧引用挖矿

**执行**:
- 如果节点落后超过 16 个 epoch：禁止挖矿
- 节点必须同步到最新 epoch 才能恢复挖矿
- 例外：开发网（允许任意引用深度用于测试）

### 4.3 孤块保留

**即时存储策略**（Epoch 级别）:
- 每个 epoch 只保留算力最大的 **16 个候选区块**
- 算力排名 17 及以后的候选区块**立即丢弃**
- 这 16 个候选区块中：1 个成为主块，15 个成为孤块

**长期保留窗口**（时间级别）: 16,384 个 epoch（约 12 天）

**理由**:
- XDAG 协议允许引用 12 天窗口内的区块
- 保留的 15 个孤块可以在 12 天内被未来区块引用
- 超过 12 天的孤块不能被新区块引用，可以安全删除
- 定期清理防止存储无限增长

**清理策略**:
- 每 100 个 epoch 运行一次（`ORPHAN_CLEANUP_INTERVAL`）
- 删除超过 16,384 个 epoch 的孤块
- 主块永不删除

**存储效果**:
- 假设 1000 个节点每 epoch 出块：1000 个候选区块产生
- 即时过滤：只保留 Top 16（丢弃 984 个）
- 12 天后清理：删除过期的孤块（保留主块）
- 存储增长：可控且可预测

## 5. 难度调整

### 5.1 目标

维持最优区块生产速率：
- **目标**: 每 epoch 150 个合格区块
- **保留**: 每 epoch 只保留算力最大的 16 个候选区块
- **效果**:
  - 控制存储增长（1000 节点 → 仅保留 16 个）
  - 确保竞争充分（150 个合格 → 选出 Top 16）
  - 维护网络去中心化（允许多节点参与）

### 5.2 调整算法

**触发条件**:
- 每 1,000 个 epoch（`DIFFICULTY_ADJUSTMENT_INTERVAL`）
- 根据最近的区块数量重新计算 `baseDifficultyTarget`

**公式**（简化版）:
```
如果 平均区块数/epoch > 目标区块数/epoch:
    baseDifficultyTarget = baseDifficultyTarget * 0.9  (更难，降低 10%)
否则如果 平均区块数/epoch < 目标区块数/epoch * 0.8:
    baseDifficultyTarget = baseDifficultyTarget * 1.1  (更容易，提高 10%)
```

**约束条件**:
- 最大调整：每次间隔 ±10%
- 最小目标：INITIAL_BASE_DIFFICULTY_TARGET
- 最大目标：2^256 - 1（实际上无难度）

### 5.3 网络算力适应

**低算力**（矿工少）:
- 每 epoch 产生的区块少
- 难度自动降低
- 保持区块链进展

**高算力**（矿工多）:
- 每 epoch 产生的区块多
- 难度自动提高
- 防止过度孤块堆积

## 6. 交易处理

### 6.1 交易生命周期

1. **提交**: 用户创建并签名交易
2. **验证**: 交易池验证签名、余额、nonce
3. **待处理**: 交易进入池中，等待纳入
4. **选择**: 矿工为区块选择高费用交易
5. **纳入**: 交易链接被添加到挖出的区块
6. **执行**: 当区块成为主块时，交易执行
7. **最终化**: 执行后交易效果永久生效

### 6.2 交易池

**选择策略**:
- 按费用排序待处理交易（降序）
- 同一账户按 nonce 排序（升序）
- 时间戳作为平局决胜（最早优先）

**限制**:
- 最大池大小：可配置（默认：10,000 笔交易）
- 每块限制：1,024 笔交易（初始保守限制）
- 未来扩展：每块最多 1,484,984 笔交易

### 6.3 交易费用

**费用计算**:
```
费用 = gasPrice * gasUsed
```

**矿工奖励**:
- 区块奖励：每块固定奖励（约每 4 年减半）
- 交易费用：区块中所有交易费用总和
- 总奖励：区块奖励 + 交易费用总和

## 7. 网络同步

### 7.1 同步策略

**完全同步**:
- 从创世块下载所有主块
- 验证每个区块的 PoW 和链接
- 执行所有交易
- 构建完整账户状态

**快速同步**（未来）:
- 下载最近的状态快照
- 仅同步最近的区块
- 新节点初始同步更快

### 7.2 区块传播

**新区块消息**:
- 矿工找到有效区块时，广播到网络
- 对等节点验证并转发给其对等节点
- 区块在几秒内传播到整个网络

**区块验证**:
1. 检查 PoW：`hash ≤ baseDifficultyTarget`
2. 验证区块结构和限制
3. 验证所有区块链接存在
4. 检查 epoch 正确性
5. 验证签名和 nonce

### 7.3 分叉解决

**场景**: 两条具有不同历史的链

**解决方案**:
```
选择累积难度最大的链
```

**过程**:
1. 节点接收竞争区块
2. 计算每条链的累积难度
3. 如有必要，切换到更重的链
4. 孤立被放弃链的区块
5. 从新主链重新执行交易

## 8. 经济模型

### 8.1 区块奖励

**初始奖励**: 取决于网络（在创世配置中设置）

**减半计划**:
- 约每 4 年减半一次
- 基于区块数量，而非时间
- 确保可预测的供应计划

### 8.2 供应动态

**总供应量**:
- 由创世配置定义
- 由于减半而接近渐近极限
- 无硬上限（可能有尾部排放）

**通胀率**:
- 由于减半而随时间递减
- 交易费用成为矿工的主要激励
- 从通胀驱动过渡到费用驱动的安全性

## 9. 安全属性

### 9.1 51% 攻击抵抗

**攻击向量**: 多数算力重写历史

**防御**:
- 累积难度使历史重写成本高昂
- 更深的区块需要指数级更多工作量
- 可以添加检查点以确保最终性

**成本**:
```
重写 N 个 epoch 的成本 = 网络算力 * N * 64 秒
```

### 9.2 双花防护

**攻击向量**: 在不同链上花费相同的币

**防御**:
- 交易仅在主链上执行
- 基于累积难度的分叉解决
- 商家应等待确认

**推荐确认数**:
- 小额交易：3-6 个 epoch（约 3-6 分钟）
- 大额交易：20-30 个 epoch（约 20-30 分钟）
- 关键交易：100+ 个 epoch（约 100+ 分钟）

### 9.3 孤块洪水

**攻击向量**: 用低质量孤块淹没网络

**防御**:
- **PoW 要求**（`baseDifficultyTarget`）限制区块生产
  - 每个区块必须满足 `hash ≤ baseDifficultyTarget`
  - 计算成本限制了攻击者生产大量区块的能力
- **Top 16 存储策略**（最强防御）
  - 每个 epoch 只保留算力最大的 16 个候选区块
  - 即使攻击者产生 10,000 个低质量区块，也只有 Top 16 被保留
  - 算力排名 17+ 的区块立即丢弃
- **孤块保留窗口**限制长期存储影响
  - 超过 12 天（16,384 epochs）的孤块自动清理
- **每个 epoch 仅引用前 16 个候选区块**
  - 新区块只引用上一 epoch 的 Top 16
  - 低质量孤块无法被未来区块引用

**效果**:
- 攻击者必须拥有超过 Top 16 的算力才能影响存储
- 存储增长完全可控：每 epoch 最多 16 个区块

## 10. 性能特征

### 10.1 当前吞吐量

**区块生产**:
- Epoch 持续时间：64 秒
- 目标候选区块数：约 150 个/epoch（通过难度调整维持）
- 保留候选区块数：Top 16/epoch（算力最大的 16 个）
- 主块数量：1 个/epoch（Top 16 中哈希值最小的）
- 有效区块速率：1 个主块 / 64 秒 = 0.016 块/秒

**交易吞吐量**:
- 每块交易数：1,024（当前限制）
- 每 epoch 主块数：1 个
- **当前 TPS**: 1,024 / 64 = **16 TPS**

**存储效率**:
- 1000 个节点出块：1000 个候选区块产生
- 存储过滤：只保留 Top 16（99.84% 丢弃）
- 每 epoch 净增：1 个主块 + 15 个孤块 = 16 个区块

### 10.2 未来扩展

**最大容量**:
- 每块链接数：1,485,000（`MAX_LINKS_PER_BLOCK`）
- 每块交易数：1,484,984（扣除 16 个区块链接后）
- **理论 TPS**: 1,484,984 / 64 = **23,203 TPS**
- **对比**: Visa 峰值容量约 24,000 TPS 的 96.7%

**扩展路径**:
1. 第一阶段：1,024 笔交易/块（16 TPS）- 当前
2. 第二阶段：10,000 笔交易/块（156 TPS）- 中期
3. 第三阶段：100,000 笔交易/块（1,562 TPS）- 长期
4. 第四阶段：1,484,984 笔交易/块（23,203 TPS）- 最终容量

### 10.3 存储需求

**区块大小**:
- 头部：约 124 字节
- 链接：每个链接 33 字节
- 1,040 个链接：每块约 34 KB
- 满容量（148.5 万链接）：每块约 48 MB

**链增长**（当前限制）:
- 每天区块数：约 1,350（每 epoch 1 块）
- 每天数据量：约 46 MB（34 KB * 1,350）
- 年增长：约 16.5 GB/年

**链增长**（满容量）:
- 每天区块数：约 1,350
- 每天数据量：约 64 GB（48 MB * 1,350）
- 年增长：约 23 TB/年

## 14. 结论

XDAG 1.0 提出了一种独特的共识机制，结合了：

1. **基于 epoch 的竞争**: 多个矿工可以在同一时间窗口内成功
2. **DAG 结构**: 孤块被保留并可以被引用
3. **PoW 安全性**: 基于工作量的链选择防止历史重写
4. **高吞吐量**: 满容量时能达到 23,203 TPS
5. **公平挖矿**: 所有有效区块都为网络安全做出贡献

该协议在保持简单优雅设计的同时，平衡了去中心化、安全性和性能。与传统区块链相比，其基于 epoch 的竞争减少了孤块浪费，其巨大的链接容量使其能够在不牺牲工作量证明共识安全属性的情况下实现 Visa 级别的交易吞吐量。

## 附录 A: 术语表

- **Epoch（时间段）**: 64 秒的区块竞争时间窗口
- **候选区块（Candidate Block）**: 在 epoch 内挖出的任何区块
- **主块（Main Block）**: epoch 竞争的胜者（最小哈希）
- **孤块（Orphan Block）**: 非胜者候选区块
- **高度（Height）**: 主链中的顺序位置
- **累积难度（Cumulative Difficulty）**: 从创世开始的难度总和
- **链接（Link）**: 对另一个区块或交易的引用
- **PoW**: 工作量证明（hash ≤ 难度目标）
- **TPS**: 每秒交易数（Transactions Per Second）

## 附录 B: 关键常量

```
EPOCH_DURATION = 64 秒
MAX_BLOCK_LINKS = 16（对上一 epoch 区块的引用）
MAX_LINKS_PER_BLOCK = 1,485,000（总容量）
MAX_TX_LINKS_PER_BLOCK = 1,024（当前保守限制）
INITIAL_BASE_DIFFICULTY_TARGET = 2^192（主网/测试网）
DEVNET_DIFFICULTY_TARGET = 2^256 - 1（无 PoW）
MINING_MAX_REFERENCE_DEPTH = 16 个 epoch（约 17 分钟）
ORPHAN_RETENTION_WINDOW = 16,384 个 epoch（约 12 天）
TARGET_BLOCKS_PER_EPOCH = 150（难度调整目标）
TOP_CANDIDATES_PER_EPOCH = 16（每 epoch 保留的候选区块数）
DIFFICULTY_ADJUSTMENT_INTERVAL = 1,000 个 epoch
ORPHAN_CLEANUP_INTERVAL = 100 个 epoch
```

**存储策略说明**:
- `TOP_CANDIDATES_PER_EPOCH = 16`: 每个 epoch 只保留算力最大的 16 个候选区块
  - 1 个成为主块（哈希值最小）
  - 15 个成为孤块（算力 Top 2-16）
  - 其余候选区块立即丢弃
- `TARGET_BLOCKS_PER_EPOCH = 150`: 难度调整目标，确保有充分的竞争
  - 150 个候选区块竞争 → 选出 Top 16 → 确保公平性
